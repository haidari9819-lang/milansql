Architecture Deep Dive

A complete tour of MilanSQL's internals — query pipeline, storage layers, optimizer design, and the engineering decisions behind a zero-dependency C++17 database engine.

⚙ Query Pipeline

Every SQL statement travels through a fixed pipeline from raw text to storage and back. Each stage is a distinct module in its own .hpp file with no circular dependencies.

SQL String ──▶ Tokenizer parser/parser.hpp · splits text into Token{type,value}
↓
SQL String ──▶ Parser parser/parser.hpp · builds ParsedCommand{type, args, clauses}
↓
SQL String ──▶ Dispatcher dispatch.hpp · routes command type → correct handler
↓
SQL String ──▶ Optimizer optimizer/ · rewrite rules, stats, join reorder, index selection
↓
SQL String ──▶ Engine engine/engine.hpp · executes DML/DDL, manages transactions
↓
SQL String ──▶ Storage storage/storage.hpp · binary format v9, WAL, buffer pool, I/O
↓
ResultSet → Dispatcher → Protocol layer → Client

🏗 System Layers

1
Client Layer

End-user interfaces: the interactive REPL, Python DB-API 2.0 driver, Node.js client, Java JDBC adapter, and any HTTP/REST client. All clients speak to the Protocol layer — none interact with the engine directly.

clients/python/ · clients/nodejs/ · REPL in main.cpp
2
Protocol Layer

Handles network I/O and wire protocol framing. TCP server on 4406 uses a simple length-prefixed text protocol. HTTP server on 8080 provides JSON REST. PG wire (5433) and MySQL wire (4407) provide standards-compatible binary protocols. GraphQL (8081) offers a query playground.

server/http_server.hpp · server/pg_server.hpp · server/mysql_server.hpp · api/graphql_server.hpp
3
Parser Layer

A hand-written recursive-descent tokenizer and parser that converts SQL text into a structured ParsedCommand object. No grammars, no ANTLR, no Bison — pure C++17. Handles the full SQL dialect including CTEs, window functions, subqueries, and MilanSQL extensions.

parser/parser.hpp · dispatch.hpp
4
Engine Layer

The execution core. Receives a ParsedCommand and executes it — DDL creates/alters schema, DML reads/modifies rows. Manages the MVCC transaction log, WAL commit protocol, savepoints, triggers, stored procedures, and row-level security policy evaluation.

engine/engine.hpp · locking/ · cdc/ · pubsub/ · replication/
5
Optimizer Layer

Sits between the Parser and Engine. The query rewriter simplifies the parsed AST before execution. The adaptive statistics module tracks column access patterns at runtime and feeds the index suggester. The parallel executor splits large scans across worker threads. The join reorderer uses cardinality estimates to choose optimal join order.

optimizer/ · parallel/ · cache/statement_cache.hpp
6
Storage Layer

Manages durable, page-based storage. Row tables use binary format v9 — a fixed-size page layout with slot arrays. Column tables use a separate columnar format. The WAL journal ensures durability: every mutation is logged before being applied. The buffer pool caches hot pages in memory. Optional compression (LZ4/RLE/Dictionary/ZSTD) operates transparently at the page level.

storage/storage.hpp · storage/column_store.hpp · compression/

🔬 Design Decisions

📦

Why header-only?

Header-only C++ eliminates the need for a separate compilation and linking step for library code. Every component can be included directly into the single main.cpp translation unit. This makes the build system trivial (CMake with one target), makes the code portable across Windows, Linux, and macOS with no platform-specific build scripts, and makes it easy for contributors to read the entire codebase in file order. The trade-off is longer compile times for a full rebuild, but incremental builds are fast because only the main translation unit needs recompiling.

🚫

Why no external dependencies?

Every external library is a potential build failure, a versioning conflict, a security vulnerability, or a licensing concern. By relying only on the C++17 standard library, MilanSQL can be cloned and compiled on any machine with a modern compiler — no apt install, no vcpkg, no Conan, no submodules. This was a deliberate pedagogical choice: if you need a feature (compression, B-trees, hash maps), you implement it yourself and understand exactly how it works.

💾

Binary format v9 — design rationale

The storage format has gone through 9 iterations (v1 was plain text, v2 added type tags, v4 introduced page-based layout, v7 added compression headers, v9 added column-store support). Version 9 uses a page-based layout with a 16-byte page header (magic number, version, row count, free-space offset), slot array for variable-length row offsets, and a data region. Each row is length-prefixed with a null bitmap for nullable columns. This layout supports efficient sequential scans, slot-level deletion without compaction, and transparent page-level compression.

🔀

Hand-written parser vs. parser generators

Tools like ANTLR or Bison would generate a parser from a grammar file, but they introduce an external dependency and a code generation step. A hand-written recursive-descent parser is pure C++, directly readable, debuggable with a standard debugger, and trivially extensible — adding a new SQL clause is a few lines of C++. The parser is also tightly coupled to the ParsedCommand output type, making it easy to add new fields without regenerating code.

⚡ Concurrency Model

MilanSQL uses multiple layers of concurrency control to support concurrent clients, parallel query execution, and background workers.

🔒 Mutexes & MVCC

A per-table std::shared_mutex protects schema and row data. MVCC (Multi-Version Concurrency Control) gives each transaction a snapshot of the database — readers never block writers and writers never block readers. Each row stores a version chain with transaction IDs.

💡 Query Cache

A global LRU query cache (configurable size) stores the result sets of identical read-only queries. Cache entries are invalidated on any write to the relevant tables. The statement cache separately stores parsed query plans to skip re-parsing identical SQL strings.

🔱 Parallel Executor

For tables exceeding the PARALLEL_THRESHOLD row count, sequential scans are split into N equal ranges and executed across a thread pool. Per-range partial results are merged via a combine function. Aggregations (SUM, COUNT, AVG) use partial aggregation with a final merge step.

📡 Pub/Sub Sessions

Each client session can LISTEN on named channels. A std::unordered_map keyed by channel name holds lists of waiting session callbacks. NOTIFY pushes a message to all registered listeners asynchronously, using a condition variable to wake blocked sessions.

📝 WAL Journal

Before any write is applied to a page, its intent is appended to the Write-Ahead Log. On crash, the WAL is replayed from the last checkpoint to restore consistency. Checkpoints flush dirty pages to disk and truncate the WAL, controlled by the background checkpoint worker.

🔗 Lock Manager

SELECT FOR UPDATE acquires a row-level exclusive lock via the lock manager. The lock manager maintains a lock table keyed by (table_id, row_id) and detects deadlocks by building a wait-for graph on each new lock request. Deadlocked transactions are aborted with a clear error message.
