B-Tree Index

MilanSQL uses a B⁺-Tree for all single-column and composite indexes. Leaf nodes store the full row key and a pointer (row offset); internal nodes store separator keys only.

Algorithm

A B-Tree of order t maintains these invariants:

  • Every non-root node has at least t−1 keys.
  • Every node has at most 2t−1 keys.
  • All leaf nodes are at the same depth.
  • Leaf nodes are linked for efficient range scans.
// Lookup: O(log n)
Node* search(Node* node, Key key) {
    int i = 0;
    while (i < node->nKeys && key > node->keys[i]) i++;
    if (i < node->nKeys && key == node->keys[i])
        return node->ptrs[i];   // found
    if (node->isLeaf) return nullptr;
    return search(node->children[i], key);
}

MVCC — Multi-Version Concurrency Control

MVCC allows readers and writers to proceed without blocking each other. Every row carries two transaction-ID fields that define its visible lifetime.

xmin / xmax Concept

struct Row {
    uint64_t xmin;  // TxID that created this row version
    uint64_t xmax;  // TxID that deleted/updated this row (0 = live)
    vector<string> values;
};

A transaction with ID txid can see a row when:

visible = (xmin <= txid) AND (xmax == 0 OR xmax > txid)

On UPDATE, the old row's xmax is set to the current TxID and a new row version is inserted with xmin = txid. DELETE only sets xmax.

Vacuum

Dead row versions (where xmax < oldest_active_txid) are reclaimed by VACUUM. This keeps the table compact without blocking ongoing queries.

// VACUUM removes rows invisible to all active transactions
VACUUM employees;
VACUUM ANALYZE employees;  -- also refreshes optimizer statistics

WAL + Double-Write Buffer

The Write-Ahead Log guarantees durability: every write is persisted to the WAL before the data page is modified. On crash, the WAL is replayed from the last checkpoint.

Log Format

// WAL entry (binary, 8-byte aligned)
struct WalEntry {
    uint64_t lsn;       // Log Sequence Number (monotonic)
    uint64_t txid;      // Transaction ID
    uint8_t  type;      // INSERT | UPDATE | DELETE | COMMIT | ROLLBACK
    uint32_t checksum;  // CRC-32 of entry payload
    uint32_t dataLen;
    uint8_t  data[];    // Serialized row delta
};

Crash Recovery

-- Manual recovery commands (Phase 114)
CHECK TABLE orders;    -- verify page checksums
REPAIR TABLE orders;   -- replay WAL, rebuild indexes

The Double-Write Buffer writes each page to a dedicated buffer area first, then to its final location. If the process crashes mid-write, recovery uses the buffer copy — preventing torn-page corruption.

Hash Join O(n+m) vs Nested Loop O(n×m)

MilanSQL's join executor selects a strategy per join pair at planning time.

Hash Join — O(n + m)

Optimal when neither side has an index on the join key.

// Build phase: hash smaller table
unordered_map<Key, vector<Row>> hashTable;
for (auto& row : smallerTable)
    hashTable[row[joinCol]].push_back(row);

// Probe phase: stream larger table
for (auto& row : largerTable) {
    auto it = hashTable.find(row[joinCol]);
    if (it != hashTable.end())
        for (auto& match : it->second)
            results.push_back(merge(row, match));
}

Nested Loop — O(n × m)

Used when both tables are small (<10 rows) or a correlated subquery forces row-by-row evaluation.

for (auto& outer : leftTable)
    for (auto& inner : rightTable)
        if (outer[joinKey] == inner[joinKey])
            results.push_back(merge(outer, inner));

DP Join-Order Planner (3+ tables)

For queries joining three or more tables, an exhaustive bitmask DP finds the optimal join order.

// cost[S] = minimum cost to join the table subset S
for (uint32_t S = 1; S < (1u << n); S++) {
    for (auto split : subsets(S)) {
        double c = cost[split] + cost[S ^ split]
                 + estimateJoin(split, S ^ split);
        if (c < cost[S]) { cost[S] = c; best[S] = split; }
    }
}

HNSW Vector Index

Hierarchical Navigable Small World (HNSW) provides approximate nearest-neighbour search in O(log n) time — used for pgvector semantic search (Phase 111).

Build Phase

Each inserted vector is placed into a random layer. Higher layers have exponentially fewer nodes, forming a coarse-to-fine navigation structure.

// Layer assignment: geometric distribution
int layer = floor(-log(random()) * mL);

// Connect to ef_construction nearest neighbours per layer
for (int l = min(layer, maxLayer); l >= 0; l--) {
    auto candidates = searchLayer(entryPoint, vec, efConstruction, l);
    connect(newNode, selectNeighbours(candidates, M));
}

Search Phase

// Greedy descent through layers, then beam search at layer 0
Node* ep = entryPoint;
for (int l = maxLayer; l > 0; l--)
    ep = greedySearch(ep, query, l);        // 1 result per layer
return beamSearch(ep, query, ef, 0);       // ef results at layer 0

DP Join-Order Optimizer

Introduced in Phase 113, the dynamic-programming planner finds the optimal join order for queries with 3 or more tables — a problem that is otherwise exponential in naive approaches.

Bitmask DP

Each subset of tables is represented as a bitmask. The planner computes the minimum-cost plan for every possible subset bottom-up.

// n = number of tables, 2ⁿ subsets
vector<double> cost(1u << n, INF);
vector<uint32_t> best(1u << n, 0);

// Base: single-table scan costs
for (int i = 0; i < n; i++)
    cost[1u << i] = scanCost(tables[i]);

// Fill: all subsets of size 2..n
for (uint32_t S = 1; S < (1u << n); S++) {
    for (uint32_t sub = (S-1)&S; sub; sub = (sub-1)&S) {
        uint32_t rest = S ^ sub;
        if (!rest || sub > rest) continue;
        double c = cost[sub] + cost[rest]
                 + joinCost(sub, rest, histograms);
        if (c < cost[S]) { cost[S] = c; best[S] = sub; }
    }
}
// Reconstruct optimal left-deep join tree
return buildPlan(best, (1u << n) - 1);
